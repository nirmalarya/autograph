Traffic Splitting:
- 0% = No traffic to canary (default)
- 5% = 5 out of 100 requests to canary
- 25% = 25 out of 100 requests to canary
- 50% = 50 out of 100 requests to canary
- 100% = All requests to canary

Metrics Monitored:
- **Error Rate**: Percentage of HTTP 5xx responses
  - Threshold: 5% (configurable)
  - Query: sum(rate(http_requests_total{status=~"5.."}[5m]))
- **P95 Latency**: 95th percentile response time
  - Threshold: 1000ms (configurable)
  - Query: histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))
- **Request Rate**: Requests per second
  - Tracked for visibility
  - Query: sum(rate(http_requests_total[1m]))

LESSONS LEARNED
================================================================================

1. Canary Deployment vs Blue-Green
   - Canary: Gradual rollout, lower risk, smaller blast radius
   - Blue-Green: All-or-nothing, faster rollout, full env testing
   - Canary better for high-traffic systems
   - Blue-Green better for database migrations
   - Both provide zero downtime
   - Both have instant rollback
   
2. NGINX Ingress Canary Annotations
   - canary: "true" enables canary routing
   - canary-weight: "0-100" sets traffic percentage
   - Atomic operation - all-or-nothing
   - Works at L7 (application layer)
   - Supports multiple routing strategies:
     * Weight-based (percentage)
     * Header-based (X-Canary: always)
     * Cookie-based (canary cookie)
   
3. Traffic Splitting Strategies
   - Start small (5%) to minimize blast radius
   - Increase gradually (5% → 25% → 50% → 100%)
   - Monitor at each stage before proceeding
   - Allow time for metrics to stabilize (60s)
   - Interactive confirmations prevent accidents
   - Always have rollback ready
   
4. Metrics Monitoring Essential
   - Error rate: Most important metric (5xx responses)
   - P95 latency: Catches performance degradation
   - Request rate: Validates traffic distribution
   - Monitor continuously during rollout
   - Set appropriate thresholds for your SLA
   - Automatic rollback on violations
   
5. Prometheus Integration
   - Query error rates: rate(http_requests_total{status=~"5.."}[5m])
   - Query latency: histogram_quantile(0.95, rate(...))
   - Query request rate: rate(http_requests_total[1m])
   - Use 5m window for stable metrics
   - Scrape interval: 15s recommended
   - ServiceMonitor for automatic discovery
   
6. Rollback Speed Critical
   - Canary rollback is instant (< 1 minute)
   - Just change Ingress canary-weight to 0
   - No pod restarts needed
   - Measure and log rollback timing
   - Verify timing meets requirements
   - Document rollback procedures
   
7. Testing Without Kubernetes
   - Validate YAML syntax with PyYAML
   - Check script syntax with bash -n
   - Simulate metrics for local testing
   - Make tests friendly for developers
   - Don't require K8s cluster for all tests
   - CI/CD can run subset of tests
   
8. Interactive Confirmations
   - Prevent accidental production changes
   - Ask before each traffic increase
   - Show current state before proceeding
   - Allow operator to review metrics
   - Builds confidence in automation
   - Can disable for CI/CD with --yes flag
   
9. Colored Terminal Output
   - Green: Success messages
   - Red: Error messages
   - Yellow: Warning messages
   - Blue: Info messages
   - Cyan: Headers
   - Improves readability significantly
   - Use ANSI color codes
   
10. Auto-Generated Documentation
    - Generate docs from test script
    - Ensures docs stay in sync with code
    - Reduces maintenance burden
    - Provides comprehensive reference
    - Run as part of test suite
    - Update docs on every change
    
11. Resource Requirements
    - Canary needs less resources than blue-green
    - 1.5x resources (stable + small canary)
    - vs 2x for blue-green (full blue + green)
    - HPA can scale stable independently
    - Canary usually 1 replica initially
    - Scale up if needed for load testing
    
12. Blast Radius Concept
    - Canary: 5% initially = limited impact
    - Blue-green: 100% immediately = full impact
    - Gradual rollout reduces risk
    - Catch issues early with minimal users
    - Increase confidence before full rollout
    - Critical for large user bases
